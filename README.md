## Модель

### Планировки

Квартира — **взвешенный граф**:
- **Вершины** = комнаты (тип, площадь, позиция центроида на сетке)
- **Рёбра** = дверные проёмы с расстоянием (евклидово расстояние между центроидами)

7 квартир охватывают типичные паттерны:

| ID | Описание | Комнат |
|---|---|:---:|
| apt_01_1br_linear | 1-комнатная, линейный коридор, 38 м² | 5 |
| apt_02_2br_corridor | 2-комнатная с центральным коридором, 55 м² | 7 |
| apt_03_studio | Студия / open-plan, 28 м² | 4 |
| apt_04_2br_study | 2-комнатная + кабинет, неудачная планировка, 65 м² | 7 |
| apt_05_3br_efficient | 3-комнатная, хабовая планировка, 80 м² | 8 |
| apt_06_1br_balcony | 1-комнатная + балкон, ванная за спальней, 44 м² | 6 |
| apt_07_2br_2bath | 2-комнатная, 2 ванные, хорошая связность, 72 м² | 7 |

### Агенты и расписание дня

#### Пожилой человек (70+)
9 переходов: спальня → кухня → ванная → гостиная → ... → спальня.  
Параметры: `speed_factor=1.5`, `transition_penalty=0.5`, усиленный штраф за длинные пути.

#### Молодая пара (партнёр 1 + партнёр 2)
7 переходов каждый, расписания сдвинуты (утренняя очередь в ванную).  
Детектируются конфликты доступа к ванной и кухне: штраф `3.0` за каждый конфликт.

#### Фрилансер (работает дома)
16 переходов, в т.ч. 5 визитов кабинет↔кухня и 2 визита кабинет↔ванная.

### Алгоритм поиска пути

Стандартный **алгоритм Дейкстры** (`simulation.py::dijkstra`).  
Входные данные: граф со взвешенными рёбрами, стартовая и конечная вершина.  
Возвращает: (суммарное расстояние, список вершин пути).

## Метрики

### Daily Distance
```
daily_distance = Σ raw_distance(trip_i)  for all trips
```
Суммарное пройденное расстояние за день (в единицах сетки планировки).

### Number of Transitions
```
n_transitions = Σ (len(path_i) - 1)  for all trips
```
Количество переходов через дверные проёмы за день.

### Average Path Length
```
avg_path_length = daily_distance / n_ok_trips
```
Средняя длина одного маршрута (только успешные поездки).

### Friction Score

Агрегированная метрика «трения», учитывающая особенности каждого агента:

```
friction = Σ_trips [ raw_distance(trip) × speed_factor ]           # базовая стоимость
         + Σ_trips [ max(0, raw_dist − threshold) × penalty_mult × speed_factor ]  # штраф за длинные пути
         + n_transitions × transition_penalty                        # штраф за частые переходы
         + n_conflicts × 3.0                                         # только для пары
```

| Параметр | Пожилой | Пара | Фрилансер |
|---|:---:|:---:|:---:|
| `speed_factor` | **1.5** | 1.0 | 1.0 |
| `long_path_threshold` | **5.0** | 6.0 | 6.0 |
| `long_path_penalty_mult` | **2.0** | 1.0 | 1.2 |
| `transition_penalty` | **0.5**/переход | 0 | 0 |
| `conflict_penalty_weight` | — | **3.0**/конфликт | — |

### Comfort Score
```
comfort_score = 100 / (1 + normalized_friction)
```
Нормировка — внутри группы агента. Чем ближе к 100 — тем удобнее.

## Ключевые выводы

1. **Центральный коридор/холл** (apt_02, apt_05) — лучший паттерн для всех агентов:
   минимальный средний путь, нет «тупиковых» цепочек.

2. **Ванная за спальней** (apt_06) — антипаттерн: +40–60% friction для пожилого, конфликты для пары.

3. **Две ванные** (apt_07) устраняют конфликты пары почти полностью.

4. **Фрилансер** получает больше пользы от близости кабинет↔кухня, чем от общего размера квартиры.

5. **Студия** (apt_03) — хороша для одного, но конфликтна для двоих.

## Ограничения

- Дискретная модель: нет учёта реального времени и длительности нахождения в комнате
- Расстояния — прямолинейные по сетке, без учёта мебели и архитектурных деталей
- Конфликты пары — по одновременности шагов расписания, без очереди ожидания
- Планировки синтетические, хотя структурно соответствуют RPLAN-формату
- Один день симуляции, без вариативности сценариев
